# СТТ · Каталог дополнительных сервисов

26 сценариев дополнительных сервисов для EDI-провайдера [СТТ](https://ctt.by/). Все решения работают через OpenAPI EDI-платформы и не требуют изменений в её коде — интеграция только «снаружи», через чтение/запись данных по API.

Для каждого сервиса: **тип выгоды**, **ICE-скор** (среднее Impact/Confidence/Ease по 10-балльной шкале), детальный функционал, техническая реализация и модель продажи клиентам.

Типы выгоды: 🟠 быстрые деньги · 🔵 lock-in · 🟣 рост сети · 🔴 стратегическая

---

## 1. Мониторинг документооборота (дашборд в реальном времени)
**Группа:** Аналитика · 🟠 Быстрые деньги · ICE 7.3

Графики трафика, ошибок и задержек по документам клиента, поверх данных из API.

**Функционал:**
- Живой граф статусов документов: отправлено / доставлено / подписано / ошибка
- Алерты по SLA — задержка подписания дольше X часов
- Разрезы по контрагенту, типу документа, филиалу
- Экспорт снапшотов для еженедельных отчётов руководству

**Как реализуется:** отдельный веб-сервис с БД для агрегатов, который раз в 1–5 минут опрашивает статусный API СТТ (или слушает вебхуки, если платформа их отдаёт) и складывает события в свою временную шкалу. Фронт — обычный дашборд с графиками, платформу не трогаем.

**Как продавать:** надстройка над любым тарифом — фиксированная плата за модуль в месяц, либо бесплатно в топовом пакете как аргумент для апсейла. Точка входа: «покажем, где теряете время на подписании».

## 2. Маршрутизация и эскалации документов
**Группа:** Операции · 🟠 Быстрые деньги · ICE 6.3

Правила распределения входящих документов по отделам, уведомления при задержках.

**Функционал:**
- Правила «документ такого типа от контрагента X → отдел Y»
- Уведомления в Telegram/почту ответственному
- Эскалация руководителю при просрочке обработки
- Журнал: кто и когда обработал документ

**Как реализуется:** rules-engine поверх вебхуков СТТ о новых входящих документах: событие приходит — сервис матчит его по правилам и дёргает интеграции уведомлений (Telegram Bot API, SMTP). Хранение правил и логов — отдельная лёгкая БД.

**Как продавать:** среднему и крупному бизнесу с несколькими отделами/филиалами через боль «документы теряются между людьми». Тариф по числу правил или пользователей.

## 3. Расширенный архив и полнотекстовый поиск
**Группа:** Доверие · 🔵 Lock-in · ICE 7.0

Поиск по пользовательским полям, экспорт в Excel поверх стандартного хранилища СТТ.

**Функционал:**
- Полнотекстовый поиск по содержимому документов
- Фильтры по произвольным полям (сумма, номенклатура, дата поставки)
- Массовый экспорт подборок в Excel/PDF
- Права доступа по ролям внутри компании клиента

**Как реализуется:** периодическая или потоковая выгрузка документов через API СТТ в собственный поисковый индекс (Elasticsearch/OpenSearch). Оригиналы и ЭЦП остаются в платформе — сервис хранит копии для удобства поиска.

**Как продавать:** бухгалтериям и юр. отделам крупных клиентов как страховка перед проверками — «найти любой документ за 10 секунд». Годовая подписка, тариф по объёму архива.

## 4. Валидатор и нормализатор документов
**Группа:** Операции · 🟠 Быстрые деньги · ICE 6.7

Проверка структуры и обязательных полей до отправки, снижает число ошибочных отправок.

**Функционал:**
- Проверка обязательных полей и структуры перед отправкой
- Сверка со справочниками 1С клиента (номенклатура, контрагенты)
- Проверка контрольных сумм и форматов
- Отчёт об ошибках с указанием, что исправить

**Как реализуется:** middleware между 1С клиента и API СТТ: документ сначала летит на валидацию (набор правил + JSON-schema), и только после успешной проверки — в платформу через штатный API.

**Как продавать:** как «страховка от штрафов и отказов» — тариф за документ или пакетами по 1000/10000 проверок в месяц. Хорошо заходит клиентам с высокой долей ручного ввода.

## 5. Шаблонизатор / конвертер форматов
**Группа:** Операции · 🟠 Быстрые деньги · ICE 6.0

Трансформация документов между внутренними форматами клиента и ЮЗДО/ТЭДО.

**Функционал:**
- Библиотека готовых маппингов между форматами клиента и ЮЗДО/ТЭДО
- Визуальный конструктор правил конвертации без программиста
- Тестовый прогон на реальных документах перед запуском

**Как реализуется:** сервис-трансформер на входе принимает файл клиента (XML/CSV/1С-выгрузка), применяет маппинг и на выходе отдаёт документ в формате, который штатно принимает API СТТ.

**Как продавать:** разовая настройка + абонплата за поддержку маппинга при изменениях форматов. Хорошо продаётся вместе с онбордингом клиентов со сложной учётной системой.

## 6. Коннекторы 1С / SAP / WMS
**Группа:** Интеграции · 🔵 Lock-in · ICE 6.3

Готовая синхронизация документов между учётной системой клиента и СТТ.

**Функционал:**
- Готовые модули синхронизации для 1С:УТ, SAP, топ-3 WMS
- Двусторонний обмен: документы туда, статусы обратно
- Мониторинг обрыва синхронизации с автоповтором

**Как реализуется:** отдельные адаптеры (плагин 1С / SAP RFC-модуль / WMS API-клиент), которые дёргают публичный API СТТ по расписанию или по событию. Ядро платформы не меняется.

**Как продавать:** разовая продажа коннектора + подписка на поддержку/обновления. Ключевой аргумент при подключении новых крупных клиентов — «не переучиваем сотрудников».

## 7. Управление контрагентами и их статусом
**Группа:** Операции · 🟠 Быстрые деньги · ICE 6.7

Реестр партнёров, автопроверка статуса в сети СТТ, регистрация новых.

**Функционал:**
- Единый реестр партнёров со статусом подключения к СТТ
- Автопроверка контрагента в сети при добавлении
- История взаимодействия и объёмов документооборота по каждому

**Как реализуется:** CRM-подобный слой поверх API СТТ: справочник контрагентов платформы синхронизируется в собственную БД раз в сутки, дополняется бизнес-метаданными (менеджер, договор, лимиты).

**Как продавать:** модуль для коммерческих и снабженческих отделов клиента. Продаём в пакете с онбордингом или отдельно тем, у кого много контрагентов и нет единого реестра.

## 8. Онбординг и миграция с других провайдеров
**Группа:** Продажи · 🟣 Рост сети · ICE 6.3

Управляемый переход с ЭДиН/podpis.by с гарантией целостности данных.

**Функционал:**
- Аудит текущего провайдера и объёма документооборота клиента
- Перенос архива и настроек в СТТ без потери истории
- Параллельный запуск и контроль расхождений в переходный период

**Как реализуется:** проектная команда использует экспорт-API старого провайдера (если доступен) и API СТТ для загрузки исторических документов и настройки маршрутизации, без вмешательства в код платформы с обеих сторон.

**Как продавать:** проектная оплата (fix или % от годового контракта), встроена в акцию «EDIальный переход» — снимает страх «а что с историей документов» и повышает конверсию с конкурентов.

## 9. Supply Chain Analytics / OTIF-мониторинг
**Группа:** Аналитика · 🟠 Быстрые деньги · ICE 7.0

Уровень сервиса, расхождения при приёмке, скорость подписания — из потока ORDERS/DESADV.

**Функционал:**
- Дашборд On-Time-In-Full по каждой сети/поставщику
- Расхождения план/факт по срокам и объёму поставки
- Сезонные паттерны заказов, ранние сигналы срыва отгрузки

**Как реализуется:** аналитический сервис читает поток ORDERS/DESADV/накладных через API СТТ, строит витрину данных (факты поставок) и считает OTIF-метрики по стандартной методологии ритейла.

**Как продавать:** поставщикам сетей как инструмент снижения штрафов за срыв поставки — подписка с ценой от объёма SKU/заказов. Верхний сегмент, но высокий чек и явный ROI.

## 10. Автосверка взаиморасчётов
**Группа:** Аналитика · 🟠 Быстрые деньги · ICE 7.3

Генерация актов сверки из фактического документооборота, подсветка расхождений.

**Функционал:**
- Автогенерация акта сверки на основе фактического документооборота
- Подсветка расхождений по суммам и датам
- Уведомление контрагента и трекинг подписания акта

**Как реализуется:** сервис агрегирует суммы по накладным/актам за период через API СТТ и сравнивает с данными учётной системы клиента, формирует акт сверки и отправляет его обратно в СТТ на подписание как обычный документ.

**Как продавать:** регулярная боль каждый квартал — продаём бухгалтерии как экономию времени. Тариф за акт или безлимитная подписка на количество контрагентов.

## 11. Скоринг платёжной и документарной дисциплины
**Группа:** Аналитика · 🔴 Стратегическая · ICE 5.0

Агрегированные индикаторы надёжности контрагента. Требует аккуратности с конфиденциальностью.

**Функционал:**
- Индекс надёжности партнёра по срокам оплаты и подписания
- Флаги аномалий (резкий рост отказов, задержек)
- Сравнение контрагента со средним по отрасли

**Как реализуется:** модель скоринга на агрегированных, обезличенных метриках из документооборота сети СТТ (не индивидуальные суммы конкретных сделок), с явным согласием участников на использование данных.

**Как продавать:** премиум-функция для крупных клиентов при выборе новых партнёров — доп. плата за отчёт или подписка. Требует юридической проработки условий использования данных до запуска.

## 12. Комплаенс-монитор (маркировка / прослеживаемость / СПОТ)
**Группа:** Комплаенс · 🔵 Lock-in · ICE 7.7

Персонализированный радар изменений законодательства под товарные группы клиента.

**Функционал:**
- Персональный радар изменений по маркировке/прослеживаемости/СПОТ под товарные группы клиента
- Чек-лист готовности к очередным нормам (например, эТрН РФ)
- Уведомления о новых требованиях с разъяснением, что делать

**Как реализуется:** контентный сервис + API-проверка: часть — экспертная аналитика законодательства (СТТ уже готовит её для сайта), часть — автоматическая проверка документов клиента через API СТТ на соответствие новым правилам.

**Как продавать:** подписка, встроенная в тарифы среднего/старшего уровня — контент для этого уже производится бесплатно для сайта, продуктизация превращает контент-маркетинг в доход.

## 13. Долгосрочный доверенный архив
**Группа:** Доверие · 🔵 Lock-in · ICE 7.3

Хранение с гарантией юридической значимости и выгрузками под проверки МНС.

**Функционал:**
- Хранение документов сверх стандартного срока платформы
- Гарантия юридической значимости на весь период хранения
- Готовые выгрузки под проверки МНС по запросу

**Как реализуется:** регулярная выгрузка подписанных документов через API СТТ в независимое защищённое хранилище с сохранением ЭЦП и метаданных, соответствующее требованиям к долгосрочному хранению.

**Как продавать:** отдельный тариф по объёму/сроку хранения. Продаём как страховку и одновременно как мощный анти-churn — чем больше архив, тем дороже клиенту уходить.

## 14. OCR-шлюз «бумага → эДокумент»
**Группа:** AI · 🟣 Рост сети · ICE 6.0

Распознавание бумажных/PDF-документов с загрузкой в СТТ через API.

**Функционал:**
- Распознавание сканов и PDF входящих документов
- Автозаполнение структурированного документа из распознанных данных
- Ручная валидация низкоуверенных полей перед отправкой

**Как реализуется:** готовый OCR-движок (тип Azure/Yandex OCR) на входе, далее маппинг распознанных полей в формат СТТ и отправка через штатный API — платформа получает уже готовый электронный документ.

**Как продавать:** тариф за страницу распознавания или подписка с лимитом. Продаём клиентам с гибридным документооборотом как мост, снижающий сопротивление партнёров переходу в ЭДО.

## 15. AI-ассистент оператора ЭДО
**Группа:** AI · 🔴 Стратегическая · ICE 6.0

Проверка накладной перед отправкой, объяснение ошибок, подсказки по маркировке.

**Функционал:**
- Проверка накладной перед отправкой на предмет типовых ошибок
- Объяснение простым языком, почему документ был отклонён
- Подсказки по правильному заполнению полей маркировки

**Как реализуется:** LLM-обёртка поверх правил валидации (сервис №4): модель получает структуру документа и текст ошибки от СТТ, формулирует понятное объяснение и рекомендацию оператору.

**Как продавать:** в составе PRO+ пакета как дифференциатор против ЭДиН/podpis.by — «умный помощник» снижает нагрузку на поддержку СТТ и повышает воспринимаемую ценность тарифа.

## 16. PIM / синхронизация каталога товаров (GTIN)
**Группа:** Данные · 🔵 Lock-in · ICE 5.7

Вынесенный сервис синхронизации товарных матриц и соответствия ePASS.

**Функционал:**
- Единый каталог товаров с GTIN, синхронизированный между поставщиком и сетями
- Контроль соответствия карточки товара требованиям сети (ePASS и др.)
- История изменений карточки и версионность

**Как реализуется:** отдельная PIM-система получает товарные атрибуты из документов клиента и через API синхронизирует изменения с профилями контрагентов в СТТ, где это применимо.

**Как продавать:** подписка по числу SKU. Продаём производителям и поставщикам, работающим одновременно с несколькими сетями — экономия на дублировании ручного ввода карточек.

## 17. Реестр ЭЦП и полномочий подписантов
**Группа:** Доверие · 🔵 Lock-in · ICE 7.7

Кто на что уполномочен, сроки действия атрибутных сертификатов, автонапоминания.

**Функционал:**
- Кто из сотрудников имеет право подписывать какие документы
- Автонапоминания об истечении сертификатов
- Аудит-трейл действий по каждому сертификату

**Как реализуется:** лёгкий административный сервис, который через API СТТ считывает список активных сертификатов компании и сопоставляет их с внутренним справочником сотрудников и их полномочий.

**Как продавать:** дёшево в реализации, высокая частота касаний — можно включать бесплатно в средний тариф как удержание, либо продавать отдельно крупным клиентам с большим штатом подписантов.

## 18. Академия ЭДО (обучение и сертификация)
**Группа:** Продажи · 🔴 Стратегическая · ICE 6.3

Сертификация специалистов клиентов и партнёров, укрепляет позицию «номер один».

**Функционал:**
- Онлайн-курсы по работе с ЭДО и требованиям маркировки
- Сертификация специалистов клиентов и партнёров-интеграторов
- Вебинары по изменениям законодательства

**Как реализуется:** обычная LMS-платформа (готовое SaaS-решение или самописное на базе открытого LMS), контент готовит СТТ, интеграция с платформой не требуется вообще.

**Как продавать:** невысокая прямая выручка, но сильный маркетинговый и лидогенерационный эффект — курсы бесплатные для привлечения, сертификация платная.

## 19. Supplier enablement как услуга
**Группа:** Рост сети · 🟣 Рост сети · ICE 7.0

Массовое подключение поставщиков крупной сети «под ключ»: обзвон, регистрация, обучение.

**Функционал:**
- Обзвон и вовлечение поставщиков сети в ЭДО
- Массовая регистрация и роуминг контрагентов
- Обучение поставщиков базовым сценариям работы

**Как реализуется:** операционная услуга (люди + скрипты + CRM для воронки), техническая часть — стандартная регистрация контрагентов через API/личный кабинет СТТ, без изменений платформы.

**Как продавать:** не поставщику, а самой сети/крупному заказчику — проектная оплата за подключение N контрагентов, часто финансируется сетью, заинтересованной в 100% охвате.

## 20. Веб-формы EDI для микропоставщиков
**Группа:** Рост сети · 🟣 Рост сети · ICE 7.3

Лёгкий портал: заказ по email-ссылке превращается в EDI-сообщение через API.

**Функционал:**
- Партнёр получает заказ по ссылке из письма — без установки ПО
- Заполняет ответ в браузере (накладная/подтверждение)
- Форма конвертируется в стандартное EDI-сообщение автоматически

**Как реализуется:** веб-приложение генерирует форму по шаблону документа, а после заполнения контрагентом трансформирует данные в EDI-сообщение и отправляет его в СТТ через API от имени зарегистрированного участника.

**Как продавать:** freemium: бесплатно для микропоставщика (барьер входа = 0), платно для инициатора (сети/поставщика, желающего охватить всех партнёров). Прямой рост числа документов в сети СТТ.
